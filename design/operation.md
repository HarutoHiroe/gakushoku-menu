# Operation

## 日常の流れ

1. `DESIGN.md` を読む
2. 正本（`design-tokens.json`）と対応表（`component-mapping.md`）を確認する
3. 外部 UI 語彙を内部パターンへ正規化してから実装する
4. 実装後に検証（drift チェック）を回す

## 検証 — drift チェック

トークンの正本（JSON）と実装側のスタイルがずれていないかを確認する。
元記事の `pnpm penpot:verify` に相当する自前チェック。最小構成では次を確認する：

- `design-tokens.json` に無い色・余白を直接ハードコードしていないか
- `component-mapping.md` の対応が実ファイルと一致しているか
- token を更新したのにスタイル側の反映が漏れていないか

> Penpot や Figma を使う場合のみ、ここにツール連携の検証を足す。
> まずは手動チェック → 慣れたらスクリプト化、で十分。

## ルール変更時

- 見た目のルールを変える → **まず `design-tokens.json`（正本）を直す** → 説明 md を更新
- 実装とルールがずれた → **正本を基準に実装を合わせる**（正本を実装に合わせない）

## つまずきポイント（元記事の教訓）

- `DESIGN.md` に詳細を詰め込むと、入口ではなく重複ドキュメントになる
- 説明 md と正本（JSON）を分けないと、実装契約が曖昧になる
- 正規化ルールが無いと、AI が毎回別の見た目を持ち込む
- verify が無いと、運用が善意依存になる
